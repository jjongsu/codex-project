# 레트로 게임 웹 프로젝트 기획서

## 1. 프로젝트 개요

### 프로젝트 목표
- `React + TypeScript + Next.js + Tailwind CSS` 기반으로 레트로 감성의 현대식 캐주얼 게임 웹 서비스를 만든다.
- 초기 버전에서는 고전 문법을 현대적으로 재해석한 게임 3종을 `/game` 라우트 하위에서 서비스한다.
- 각 게임은 플레이 후 점수를 등록할 수 있으며, 점수 등록 이름은 `영문 3글자`까지만 허용한다.
- 백엔드는 `Supabase`를 사용하여 점수 저장과 랭킹 조회를 처리한다.
- 배포는 `Vercel`을 사용한다.
- 장기적으로 `Google AdSense`를 통한 수익화를 목표로 하며, 광고 외에도 검색 유입이 가능한 콘텐츠를 함께 운영한다.

### 프로젝트 방향성
- 누구나 접속해서 바로 플레이할 수 있는 가벼운 웹 게임 경험
- 로그인 없이도 랭킹 등록이 가능한 단순한 구조
- 모바일과 PC 모두에서 플레이 가능한 반응형 UI
- 게임만 있는 단순 사이트가 아니라, 향후 콘텐츠 확장이 가능한 게임 포털 형태

## 2. 1차 출시 범위

### 제외 범위

## 3. 게임 선정안

초기 3개 게임은 최신 브라우저/캐주얼 트렌드, 모바일 적합성, 구현 난이도, 점수 경쟁성을 함께 고려하여 아래처럼 확정한다.

1. `Block Jam Blitz`
2. `Snake Survivor`
3. `Brick Shot Rush`

선정 이유:
- `Block Jam Blitz`는 모바일 친화적인 배치형 퍼즐로 진입장벽이 낮고 재시도성이 높다.
- `Snake Survivor`는 고전 스네이크 인지도를 유지하면서도 생존과 강화 선택으로 현대식 액션 흐름에 맞춘다.
- `Brick Shot Rush`는 클래식 `Breakout`보다 모바일 조작과 물리적 손맛이 좋은 각도 발사형 구조다.
- 세 게임 모두 짧은 세션, 높은 재도전성, 랭킹 경쟁, 향후 일일 챌린지 확장성이 좋다.

개발 참고 문서:
- [트렌드 맞춤 3종 추천안](./trend-fit-game-lineup.md)
- [트렌드 맞춤 게임 상세 기획서](./trend-fit-game-design-briefs.md)

주의 메모:
- `Block Jam Blitz`는 `Tetris`와 혼동될 정도로 유사한 명칭, 룰, 비주얼을 피한다.
- 순수 `Breakout` 복각은 보류하고, `Brick Shot Rush`처럼 모바일 우선 구조로 운영한다.
- 세부 룰, 점수 규칙, MVP 범위는 위 참고 문서를 기준으로 고정한다.

## 5. 핵심 기능 정의

### 게임 플레이
- 페이지 접속 시 바로 플레이 가능해야 한다.
- 시작, 일시정지, 재시작이 가능해야 한다.
- 게임 오버 시 점수 등록 UI를 노출한다.
- 모바일에서는 터치 UI 또는 간단한 조작 버튼을 제공할 수 있다.

### 점수 등록
- 플레이어 이름은 `A-Z 영문 3글자`까지만 허용한다.
- 예시: `AAA`, `KIM`, `JAY`
- 소문자 입력 시 자동으로 대문자로 변환한다.
- 1차 버전의 저장 정책은 아래처럼 고정한다.
  - `1~3글자` 허용
  - 저장 시 공백 제거
  - 특수문자 및 한글 차단

### 랭킹
- 게임별 Top N 랭킹 조회
- 최신 점수 등록 직후 랭킹 반영
- 동일 이름 중복 허용
- 필요 시 `최고 점수 기준`, `최신순`, `오늘의 점수` 필터로 확장 가능

## 6. 백엔드 설계 초안

### Supabase를 선택하는 이유
- 백엔드 지식이 많지 않아도 시작하기 쉽다.
- 데이터베이스(PostgreSQL), API, 인증, 배포형 관리 화면을 한 번에 제공한다.
- Next.js와 연동 자료가 많다.
- 초기 게임 랭킹 저장 기능 정도는 비교적 간단하게 구현 가능하다.

### 1차 버전에서 필요한 백엔드 범위
- 점수 저장
- 점수 조회
- 기본적인 입력 검증
- 제출 빈도 제한
- 비정상 점수 차단을 위한 최소한의 보호 로직

### 백엔드 최종 판단
- 현재 프로젝트 범위에서는 `Supabase + Next.js Route Handler + Vercel` 조합으로 충분히 문제없이 구성 가능하다.
- 다만 아래 조건을 지키지 않으면 운영 단계에서 보안 또는 구조 문제가 생길 수 있다.
  - 브라우저에서 점수 테이블에 직접 쓰지 않기
  - RLS 활성화 유지
  - 서버 전용 키를 브라우저에 절대 노출하지 않기
  - 점수 검증을 DB와 서버 양쪽에서 수행하기
  - rate limit을 메모리 기반이 아닌 외부 또는 플랫폼 기능으로 처리하기

### 권장 백엔드 구조
- 브라우저 -> `Next.js Route Handler`
- Route Handler -> `Supabase`
- 점수 저장과 랭킹 조회 모두 일단 Route Handler를 경유한다.
- 이렇게 하면 추후 캐싱, 검증, 필터링, abuse 대응을 한 곳에서 관리하기 쉽다.

### 인증 전략
- 1차 버전에서는 `Supabase Auth`를 도입하지 않는다.
- 이유:
  - 현재 요구사항은 로그인 없이 점수 저장이 가능해야 한다.
  - Auth를 넣으면 사용자 세션, 쿠키, 정책 설계가 불필요하게 복잡해진다.
- 따라서 초기에는 `익명 점수 저장 API`만 운영한다.
- 추후 회원 기능이 필요할 때만 `Supabase Auth`를 추가한다.

### 추천 데이터 구조

#### `high_scores` 테이블
- `id`: uuid
- `game_slug`: text
- `player_name`: text
- `score`: integer
- `created_at`: timestamp

### 권장 컬럼 보강
- `id`: `uuid primary key default gen_random_uuid()`
- `game_slug`: `text not null`
- `player_name`: `text not null`
- `score`: `integer not null`
- `created_at`: `timestamptz not null default now()`

### 권장 DB 제약조건
- `game_slug` 체크 제약조건
  - 허용값: `block-jam-blitz`, `snake-survivor`, `brick-shot-rush`
- `player_name` 체크 제약조건
  - 정규식: 영문 대문자 `1~3글자`
- `score` 체크 제약조건
  - `0 이상`

### 권장 인덱스
- `(game_slug, score desc, created_at asc)` 복합 인덱스
- 목적:
  - 게임별 랭킹 조회 최적화
  - 동점 시 정렬 기준 고정

### 저장 규칙
- `game_slug`는 `block-jam-blitz`, `snake-survivor`, `brick-shot-rush` 중 하나
- `player_name`은 영문 대문자 1~3글자만 허용
- `score`는 0 이상 정수

### 1차 보호 로직
- 동일 세션 또는 IP 기준으로 너무 짧은 간격의 반복 제출은 제한한다.
- 게임별 이론상 비정상적으로 높은 점수는 서버에서 한 번 더 검토한다.
- 클라이언트가 보내는 점수는 신뢰하지 않고, 최소한의 서버 검증을 거친다.
- Vercel 메모리 내부 변수만으로 rate limit을 구현하지 않는다.
- 우선순위는 아래처럼 둔다.
  - 1안: Vercel WAF Rate Limiting
  - 2안: Redis 기반 rate limit
  - 3안: 임시로 DB 기반 제한

### 안티치트 현실 판단
- 브라우저에서 실행되는 게임 특성상, 점수 조작을 `완전하게` 막는 것은 어렵다.
- 1차 버전의 목표는 `완전 차단`이 아니라 `남용 최소화`로 둔다.
- 즉, 아래 수준을 목표로 한다.
  - 무작위 스팸 제출 방지
  - 형식이 잘못된 데이터 차단
  - 명백히 비정상적인 점수 차단
- 강한 수준의 치트 방지가 필요하면 추후 아래 같은 별도 설계를 검토한다.
  - 서버 검증 가능한 게임 이벤트 로그
  - 리플레이 데이터 검증
  - 계정 기반 기록 관리

### Supabase 정책 방향
- 초기에는 로그인 없이 점수 저장
- 다만, 아무 값이나 들어가지 않도록 입력 검증 필요
- 추후 보안 강화를 위해 아래를 검토
  - Row Level Security 정책
  - 서버 경유 저장
  - rate limit 성격의 보호 장치

### RLS 원칙
- `high_scores` 테이블은 반드시 `RLS enabled` 상태로 유지한다.
- 1차 버전에서는 브라우저에서 직접 테이블 접근을 허용하지 않는 방향을 기본값으로 둔다.
- 즉, `anon` 또는 `authenticated`용 점수 `insert` 정책은 일단 만들지 않는다.
- 서버 Route Handler에서만 점수 저장을 수행하고, 서버 전용 키로 처리한다.
- 이렇게 하면 실수로 클라이언트가 직접 DB에 쓰는 상황을 줄일 수 있다.

### 구현 권장 방식
- 클라이언트에서 바로 Supabase에 쓰기보다, `Next.js Route Handler`를 통해 점수를 저장하는 방식 권장
- 이유:
  - 검증 로직을 서버 쪽에 둘 수 있다.
  - 향후 보안 강화가 쉽다.
  - Supabase를 "백엔드처럼" 간단히 사용할 수 있다.
- 환경변수 원칙:
  - 브라우저 공개용: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_DEFAULT_KEY`
  - 서버 전용: `SUPABASE_SECRET_KEY`
  - 서버 전용 키는 브라우저 번들에 절대 포함되지 않도록 한다.

### Supabase 클라이언트 구성 원칙
- 브라우저용 Supabase 클라이언트와 서버용 Supabase 클라이언트를 분리한다.
- 서버 전용 키를 사용하는 클라이언트는 `SSR용 사용자 세션 클라이언트`와 섞지 않는다.
- 특히 `service role` 또는 `secret key` 성격의 서버 전용 키는 사용자 세션/쿠키를 공유하는 SSR 클라이언트와 혼용하지 않는다.
- 점수 저장용 서버 클라이언트는 Route Handler 내부 또는 서버 전용 유틸에서 `supabase-js`로 분리 생성한다.

### Route Handler 권장 역할
- `POST /api/scores`
  - 입력 검증
  - rate limit 확인
  - 서버 측 점수 검증
  - Supabase insert
- `GET /api/scores?game=block-jam-blitz`
  - 게임별 Top N 랭킹 조회
  - 정렬 고정
  - 필요 시 캐싱 적용

### 런타임 권장사항
- 점수 API는 `Node.js runtime` 기준으로 구현한다.
- 초기에는 Edge Runtime을 굳이 사용하지 않는다.
- 이유:
  - 디버깅이 단순하다.
  - 서버 전용 환경변수 관리가 더 직관적이다.
  - 현재 기능 범위에서 Edge의 장점이 크지 않다.

### 인프라 권장사항
- Supabase 프로젝트 리전과 Vercel 배포 리전을 최대한 가깝게 맞춘다.
- 랭킹 API는 조회 빈도가 높아질 수 있으므로 추후 캐싱 적용 여지를 남긴다.

## 7. 프론트엔드 구조 방향

### 기술 스택
- `Next.js App Router`
- `TypeScript`
- `Tailwind CSS`
- `Phaser`
- `Zustand`

### 게임 렌더링 원칙
- 게임 루프와 충돌 계산은 React 렌더링 흐름에 의존하지 않는다.
- React는 페이지, HUD, 모달, 랭킹 UI를 담당하고, 실제 게임 업데이트는 `Phaser Scene`에서 처리한다.
- 즉, `React는 UI 셸`, `Phaser는 게임 실행 엔진`, `Zustand는 공통 상태 저장소`로 역할을 분리한다.

### 상태 관리 원칙
- 게임 내부 프레임 단위 상태는 Phaser 내부에서 처리한다.
- UI에 보여줄 상태만 `Zustand`로 올린다.
- 예시:
  - 현재 점수
  - 최고 점수
  - 일시정지 여부
  - 음소거 여부
  - 게임 오버 상태
  - 점수 제출 모달 오픈 여부

### 공통 컴포넌트 후보
- `Header`
- `Footer`
- `GameCard`
- `GameLayout`
- `ScoreBoard`
- `ScoreSubmitForm`
- `Modal`
- `AdSlot`
- `ContentCard`

### 게임 공통 규칙
- 게임 렌더링 영역과 UI 영역을 분리
- 점수 계산 로직을 게임별로 분리
- 랭킹 조회 및 제출 UI는 공통 컴포넌트화
- Phaser 인스턴스 생성/정리 로직은 공통 래퍼로 관리

## 8. 점수 등록 UX 기획

### 사용자 흐름
1. 사용자가 게임 플레이
2. 게임 오버
3. 점수 결과 표시
4. 이름 3글자 입력
5. 점수 저장
6. 랭킹 갱신 결과 표시

### 입력 UX 원칙
- 입력 칸은 3칸 또는 단일 입력칸 중 하나로 구현
- 자동 대문자 변환
- 영문 외 입력 차단
- 저장 중/성공/실패 상태 표시

### 예외 처리
- 이름 미입력 시 저장 불가
- 서버 오류 시 재시도 안내
- 너무 빠른 반복 저장은 제한 고려

## 18. 게임 개발 기술 선택 가이드

### 결론
- 현재 프로젝트는 `Phaser 도입`을 기준으로 진행한다.
- 이유는 게임이 3종 이상으로 확장될 예정이고, 씬 관리와 입력 처리, 오디오, 이펙트, 공통 패턴을 초기에 정리해두는 편이 이후 유지보수에 유리하기 때문이다.
- 상태 관리는 `Zustand`를 함께 사용하되, Phaser 내부 실시간 상태 전체를 저장하지 않고 UI와 공유할 최소 상태만 관리한다.

### 선택안: `Phaser + Zustand`

구성:
- 게임 렌더링: `Phaser`
- 게임 루프: `Phaser Scene`
- 입력 처리: `Phaser Input`
- UI: `React + Tailwind CSS`
- 공통 상태: `Zustand`
- 점수 저장: `Next.js Route Handler + Supabase`

장점:
- 씬, 입력, 사운드, 애니메이션 구조를 일관되게 가져갈 수 있다.
- 게임이 늘어나도 공통 패턴을 유지하기 쉽다.
- `Block Jam Blitz`, `Snake Survivor`, `Brick Shot Rush` 모두에 잘 맞는다.
- React와 게임 엔진의 역할 분리가 분명해진다.

단점:
- 초기 학습 비용이 있다.
- Next.js/React와 Phaser의 라이프사이클 경계를 처음에 정리해야 한다.
- 상태를 Phaser와 Zustand에 이중으로 잘못 저장하면 구조가 복잡해질 수 있다.

적합한 경우:
- 게임이 3개 이상으로 늘어날 계획이 분명한 경우
- 공통 게임 프레임워크를 초기에 잡고 싶은 경우
- 이후 이펙트, 사운드, 연출 확장을 고려하는 경우

### 권장 아키텍처

- `React`
  - 페이지 라우팅
  - 점수판 HUD
  - 모달
  - 랭킹 UI
  - 광고 및 콘텐츠 영역
- `Phaser`
  - 게임 씬
  - 충돌 처리
  - 입력 처리
  - 오디오
  - 실제 렌더링
- `Zustand`
  - UI에 공유할 최소 상태
  - 설정값
  - 게임 세션 상태

### Zustand에 넣을 상태 예시
- `score`
- `bestScore`
- `isPaused`
- `isMuted`
- `isGameOver`
- `playerNameDraft`
- `submitStatus`

### Phaser 내부에 둘 상태 예시
- 플레이어 좌표
- 블록 위치
- 충돌 계산 중간값
- 프레임 단위 속도/가속 값
- 적, 공, 브릭 등의 실시간 엔티티 상태

### 대안: 엔진 없이 개발

장점:
- 의존성이 적다.
- 게임 로직을 더 직접적으로 이해할 수 있다.

단점:
- 게임 수가 늘수록 공통 엔진성 코드를 직접 관리해야 한다.
- 씬과 오디오, 입력 처리 구조를 매번 더 많이 직접 구성해야 한다.

### 보조 선택지: `PixiJS`

`PixiJS`는 공식 사이트에서 빠르고 유연한 2D 렌더러라고 설명하고, FAQ와 가이드에서는 `게임 엔진이 아니라 렌더링 엔진`이라는 점을 분명히 한다.

판단:
- 성능 좋은 2D 렌더링이 필요할 때는 좋다.
- 하지만 현재 프로젝트에서는 `Phaser`보다 직접 구성할 것이 많고, `Canvas 직접 구현`보다도 구조가 애매해질 수 있다.
- 따라서 이번 프로젝트 1차에서는 우선순위가 높지 않다.

### 함께 쓰기 좋은 작은 라이브러리

필수에 가까움:
- `@supabase/supabase-js`: Supabase 연동
- `zod`: 점수 등록 API 입력 검증

권장:
- `howler.js`: 게임 효과음/배경음 제어
- `clsx` 또는 `cn` 유틸: UI 클래스 조합 정리

현재는 비권장:
- 무거운 물리 엔진
- 복잡한 애니메이션 프레임워크
- 랭킹용 상태관리 대형 라이브러리
